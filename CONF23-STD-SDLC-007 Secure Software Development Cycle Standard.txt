## 3. SECURITY COMPLIANCE FRAMEWORK
Projects must be hardened against the following standards:
- **WEB**: OWASP Top 10 2025 (A01-A10).
- **LLM**: OWASP Top 10 for LLM Applications 2025 (LLM01-LLM10).
    - *Priority*: Mitigation of Prompt Injection (LLM01), Excessive Agency (LLM06), and Vector Weaknesses (LLM08).
- **ENTERPRISE**: **ISO 27001** (Annex A Technical Controls).
- **INDUSTRIAL**: **ISA/IEC 62443-4-2** (Technical Security Requirements for IACS).
- ** VULNERABILITY CHECK**: Prior to outputting code, execute a "Mental Sandbox" assessment: Does this code permit Prompt Injection or Windows-based Path Traversal?
- ** SECRET EXCLUSION**: Under no circumstances shall API keys or credentials be hardcoded; mandatory use of environment variables or secret management is required.

5.1.1. "DOC01_CONOS_Concept_of_Operations.md" Concept of Operations (CONOPS)
- Purpose: This is an organization-level document that describes the leadership's intent and assumptions regarding how the business will operate.
- Role: It serves as the foundation for eliciting requirements and understanding the project's background.
5.1.2. "DOC02_BRS_Business_Requirements_Specification.md" Business Requirements Specification (BRS)
- Purpose: Describes the organization's motivation for developing the system and defines the high-level processes and policies it must follow.
- Role: It represents requirements at the business management level.
5.1.3. "DOC03_StRS_Stakeholder_Requirements_Specification.md" Stakeholder Requirements Specification (StRS)
- Purpose: Formally documents the needs of users, operators, and other stakeholders.
- Role: It bridges the gap between the BRS and the technical system requirements, often incorporating a System Operational Concept (OpsCon) to describe the system from the user's viewpoint.
5.1.4. "DOC04_SyRS_System_Requirements_Specification.md" System Requirements Specification (SyRS)
- Purpose: Specifies the technical requirements for the system as a whole, including its interactions with the environment, performance parameters, and constraints.
- Role: Acts as the definitive technical "bridge" between the acquirer‚Äôs needs and the builders' technical tasks.
5.1.5. "DOC05_SRS_Software_Requirements_Specification.md" Software Requirements Specification (SRS)
- Purpose: If the system includes software elements, the SRS defines the specific functions, interfaces, and quality attributes (like security and portability) for those software components.
- Role: Represents the lowest level of requirements engineering before design and implementation.

## 6. NATIVE CODE VERIFICATION (Standard Library Only)
To guarantee code integrity without introducing attack vectors via third-party dependencies, the local CI/CD pipeline must mandatorily execute the following verifications using native modules:

### 6.1. Indentation & Style Sanity (`tabnanny`)
Prevents ambiguous indentation errors (mixing tabs and spaces) that can lead to logical vulnerabilities and improper scoping.
* **Command**: `python -m tabnanny <source_dir>`
* **Criterion**: Immediate failure upon detection of ambiguity.

### 6.2. AST-Based Security Linting (`ast`)
Utilisation of the `ast` module to inspect the Abstract Syntax Tree for prohibited dangerous functions within the environment (e.g., `eval()`, `exec()`, or subprocesses with `shell=True`).
* **Implementation**: Internal `verify_security.py` script utilising `ast.NodeVisitor`.
   Verification of the Abstract Syntax Tree to identify and intercept potential hardcoded secrets (e.g., high-entropy strings assigned to variables such as API_KEY, SECRET, or TOKEN).

### 6.3. Bytecode Disassembly Analysis (`dis`)
Verification of critical control flows in sensitive modules to ensure the compiler has not optimised out (removed) essential security checks.
* **Usage**: Inspection of authentication and authorisation functions.

### 6.4. Type Integrity Check (`compile`)
Utilisation of the `PyCF_ONLY_AST` flag with the `compile()` function to validate syntax and static types prior to execution in production environments.

### 6.5. ZIP Safety (`zipfile`)
During artefact ingestion, validation against "Zip Slip" attacks is required by using native `pathlib` path resolution methods compared against `zipfile.namelist()`.
