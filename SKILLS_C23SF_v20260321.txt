## Phase 0: The Mindset Shift
An AI agent is not just a chatbot. It’s a system that:
- Receives goals
- Reasons/plans
- Uses tools
- Remembers context
- Takes actions
- Handles failures
- Gets evaluated and improved
Think of it like building a small software product with an LLM at the center.

## Phase 1: Define Purpose, Scope, and Success Criteria
- **1.1 Define the exact job of the agent**: Be specific. (e.g., “Customer support agent for billing questions”).
- **1.2 Define user types**: Internal teams, External customers, Technical vs non-technical users.
- **1.3 Define use cases (primary + secondary)**: List real tasks. Define "Out of scope" (explicitly forbidden).
- **1.4 Define success criteria (measurable)**: Task completion rate, Accuracy, Hallucination rate, Tool success rate, Latency, Cost.
- **1.5 Define constraints**: Compliance/privacy (PII, HIPAA, GDPR), Allowed tools/APIs, Budget, Max response time, Security boundaries.

## Phase 2: Design the Agent Behavior (System Prompt + Policies)
- **2.1 Define the agent role/persona**: Operational role defining tone, depth, risk tolerance, and allowed actions.
- **2.2 Define goals and priorities**: Explicit priority order for operations.
- **2.3 Write operational instructions (System Prompt)**: Mission, Tool usage policy, Output schema (JSON/Markdown), Error handling, Escalation, and Guardrails.
- **2.4 Add “thinking structure”**: Define hidden structure: Understand -> Check tools -> Retrieve context -> Plan action -> Execute -> Validate -> Return result.
- **2.5 Add examples (few-shot)**: Examples of good behavior, edge cases, refusal cases, and escalation cases.

## Phase 3: Choose the LLM and Model Strategy
- **3.1 Choose by task type**: Fast/cheap (classification/routing), Reasoning (planning), Long-context (large docs), Specialized (coding).
- **3.2 Tune parameters**: Temperature, Max tokens, Top-p.
- **3.3 Decide fallback strategy**: Fallback models for timeouts or high costs.
- **3.4 Cost and latency planning**: Estimated tokens per task, Volume, and SLA targets.

## Phase 4: Tooling & Integrations
- **4.1 Design tool categories**: Knowledge (Web search, RAG, DB), Action (Ticket, CRM, Workflow), Computation, Communication.
- **4.2 Tool interface design**: Clear name, structured inputs/outputs, error codes, and auth models.
- **4.3 Use function-calling schemas**: Strict JSON schemas for reliability and validation.
- **4.4 Add MCP / tool server layer**: Use standard interfaces (like Model Context Protocol) to connect to files, browser, and enterprise systems.
- **4.5 Add “agent as a tool”**: Coordinator pattern with one coordinator and multiple specialist agents as tools.

## Phase 5: Memory Design (Basic)
- **5.1 Working memory (short-term)**: Current request, plan, and intermediate results.
- **5.2 Conversation memory (episodic)**: Multi-turn chat history (use summarization to prevent bloat).
- **5.3 Long-term memory**: Persistent user profiles, preferences, and recurring workflows (Structured DB).
- **5.4 Retrieval memory (RAG)**: Chunking, embedding, retrieving, and reranking documentation.

## Phase 6: Orchestration & Runtime
- **Goal**: Build the backbone of the agent's logic to handle complex, asynchronous actions.
- **Actions**: Implement state management, triggers, and retry logic to handle tool calls and API timeouts.
- **Deliverables**: Runtime flow diagram and state machine.

## Phase 7: Memory Design (Advanced)
- **Goal**: Architect a sophisticated system for data persistence and retrieval.
- **Actions**: Refine the multi-layer memory system (short-term, episodic, and long-term RAG/Vector storage). Prevent “memory pollution” via confidence thresholds.
- **Deliverables**: Finalized Memory architecture and retention policy.

## Phase 8: UI & Delivery Layer
- **Goal**: Optimize how users interact with the agent.
- **Actions**: Design the UX flow and include Human-in-the-Loop (HITL) checkpoints for high-risk actions.
- **Deliverables**: UX flow and API contract.

## Phase 9: Testing & Evaluations (Evals)
- **Goal**: Ensure reliability and prevent regressions.
- **Actions**: Create an evaluation dataset to track tool correctness, hallucination rates, and performance over time.
- **Deliverables**: Eval suite and metrics dashboard.

## Phase 10: Safety & Governance
- **Goal**: Secure the agent against vulnerabilities and ensure compliance.
- **Actions**: Implement prompt injection defenses, least-privilege permissions for tools, and comprehensive audit logging.
- **Deliverables**: Security checklist and approval matrix.

## Phase 11: Deployment & Iteration
- **Goal**: Successfully launch and maintain the agent in production.
- **Actions**: Establish staging environments and real-time observability to monitor token usage, latency, and failure rates.
- **Deliverables**: Deployment plan and monitoring strategy.

## 6. CODE VERIFICATION
To guarantee code integrity without introducing attack vectors via third-party dependencies, the local CI/CD pipeline must mandatorily execute the following verifications using native modules:

REQ6.01 - [X] Indentation & Style Sanity (`tabnanny`)
           Prevents ambiguous indentation errors (mixing tabs and spaces) that can lead to logical vulnerabilities and improper scoping.
           * **Command**: `python -m tabnanny <source_dir>`
           * **Criterion**: Immediate failure upon detection of ambiguity.
REQ6.02 - [X] AST-Based Security Linting (`ast`)
          Utilisation of the `ast` module to inspect the Abstract Syntax Tree for prohibited dangerous functions within the environment (e.g., `eval()`, `exec()`, or subprocesses with `shell=True`).
          * **Implementation**: Internal `verify_security.py` script utilising `ast.NodeVisitor`.
          * Verification of the Abstract Syntax Tree to identify and intercept potential hardcoded secrets (e.g., high-entropy strings assigned to variables such as API_KEY, SECRET, or TOKEN).
REQ6.03 - [X] Bytecode Disassembly Analysis (`dis`)
          Verification of critical control flows in sensitive modules to ensure the compiler has not optimised out (removed) essential security checks.
          * **Usage**: Inspection of authentication and authorisation functions.
REQ6.04 - [X] Type Integrity Check (`compile`)
          Utilisation of the `PyCF_ONLY_AST` flag with the `compile()` function to validate syntax and static types prior to execution in production environments.
REQ6.05 - [X] ZIP Safety (`zipfile`)
